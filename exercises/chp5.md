# 5장 연습문제

## 퍼즐

### 가로

**1 요구 분석 내용 가운데 ER 다이어그램이나 트랜잭션 명세서에 표현하기 어려운 운영상의 제약사항**
- 답: **업무규칙**

**3 DB에서 하나의 논리적 기능을 수행하기 위한 작업의 기본 단위**
- 답: **트랜잭션**

**4 데이터 요구 분석 명세서를 기초로 하여 DBMS와는 독립적인 개념적 데이터 모델, 즉 개념적 스키마를 표현하는 것**
- 답: **개념적스키마모델링**

**8 개체나 관계의 특성을 나타내는 요소**
- 답: **속성**

**9 피터 첸 표기법에서 속성을 표현하는 기호**
- 답: **타원형**

### 세로

**2 트랜잭션 요구 분석 명세서를 기초로 하여 업무 단위별로 유형별 트랜잭션을 표현하는 것**
- 답: **트랜잭션모델링**

**4 DB가 표현하려고 하는 유·무형의 정보 대상으로서 현실 세계에 존재하면서 서로 구별될 수 있는 요소**
- 답: **개체**

**5 피터 첸 표기법에서 개체를 표현하는 기호**
- 답: **직사각형**

**6 개체 간의 의미 있는 연결 또는 연관성을 의미하는 요소**
- 답: **관계**

**7 피터 첸 표기법에서 관계를 표현하는 기호**
- 답: **마름모**

## 연습문제

### 1. 개념적 설계 단계의 두 가지 주요 업무가 무엇인지 설명하시오.
- 답: **개념적 스키마 모델링**과 **트랜잭션 모델링**이다.
- 설명:
  - 개념적 스키마 모델링: 데이터 요구 분석 명세서를 바탕으로 DBMS와 독립적인 개념적 데이터 모델, 즉 개념적 스키마를 표현하는 작업
  - 트랜잭션 모델링: 트랜잭션 요구 분석 명세서를 바탕으로 업무 단위별로 유형별 트랜잭션을 표현하는 작업

### 2. ER 모델을 활용해서 개념적 스키마 모델링을 수행하는 순서를 바르게 나열하시오.
1. 개체 타입의 속성을 식별한다.
2. 개체 타입 간의 관계 타입을 식별한다.
3. 관계 타입의 속성을 식별한다.
4. 관계 타입의 유형과 카디널리티(옵션)를 결정한다.
5. 데이터 요구 분석 명세서를 기초로 하여 핵심 개체 타입을 식별한다.
6. 개체 타입의 주 식별자, 즉 기본키 속성을 결정한다.
7. ER 다이어그램이 데이터 요구 분석 명세서에 포함된 모든 요구사항을 반영하는지 검증한다.
8. 식별한 개체 타입과 관계 타입, 개체 및 관계 타입의 속성을 ER 다이어그램으로 표현한다.

- 답: **5 → 2 → 4 → 1 → 6 → 3 → 8 → 7**

### 3. 다음 중 개체 식별 방법에 대한 설명이 잘못된 것은?
1. 데이터 요구 분석 명세서에서 주로 주어나 목적어로 표현된 것이 개체 후보가 된다.
2. 개체 후보를 검토하면서 개체 이름을 변경할 수 있다.
3. 실제 데이터 수가 2개 이상 존재해야 최종 개체가 된다.
4. 동일한 의미를 갖는 개체도 이름이 다르면 최종 개체가 된다.

- 답: **4번**
- 설명: 동일한 의미를 갖는 개체는 이름이 달라도 중복 개체로 보고 제외한다.
- 보충: 교안 설명을 엄밀히 적용하면 3번도 예외가 있다. 현재 데이터가 1개뿐이어도 추후 추가 가능성이 있으면 최종 개체가 될 수 있다.

### 4. 다음 중 개체 이름을 결정할 때의 유의 사항이 잘못 설명된 것은?
1. 가능하면 약어를 사용해서 간단히 표현한다.
2. 이름의 일관성을 위해 이름 부여 규칙을 미리 정하는 것이 좋다.
3. 단수 명사를 사용한다.
4. 현업에서 익숙한 표준용어를 사용한다.

- 답: **1번**
- 설명: 교안은 가능하면 약어를 사용하지 않는 것이 좋다고 설명한다.

### 5. 다음 중 관계 식별과 관련된 설명이 잘못된 것은?
1. 데이터 요구 분석 명세서에서 주로 동사로 표현된 부분을 관계로 식별한다.
2. 식별된 개체 가운데 관계로 표현하는 것이 적합한 것은 관계로 전환한다.
3. 관계는 개체와 관련된 업무수행 규칙에 해당한다.
4. 한 번 설정된 관계는 변경하기 어려우므로 신중히 결정해야 한다.

- 답: **4번**
- 설명: 교안은 업무 규칙이 변경됨에 따라 관계도 변경될 수 있음을 염두에 두라고 설명한다.

### 6. 다음 중 속성 식별과 관련된 설명이 잘못된 것은?
1. 데이터 요구 분석 명세서에서 개체나 관계를 설명하는 명사를 개체의 속성으로 식별한다.
2. 데이터 요구 분석 명세서에서 개체의 기본키 속성을 식별할 수 없을 때는 기본키 역할을 할 속성을 추가한다.
3. 개념적 설계 단계에서는 주요 속성만 식별한다.
4. 필요한 경우 속성 이름을 변경할 수 있다.

- 답: **1번**
- 설명: 개체를 설명하는 명사는 개체의 속성으로, 관계를 설명하는 명사는 관계의 속성으로 식별해야 한다.
- 보충: 3번도 교안과 완전히 같지는 않다. 교안은 개념적 설계 단계에서 가능한 모든 속성을 식별하라고 설명한다.

### 7. (____) 안에 적합한 단어는?
> 피터 젠 표기법에서 관계는 1개 이상의 개체를 실선으로 연결하는 (____)로 표현하고, (____)을 실선 위에 표시한다.

- 답: **마름모, 관계 유형**

### 8. 다음 중 관계 이름을 결정할 때의 유의 사항이 잘못 설명된 것은?
1. 개체 간의 관련성이 명확하도록 표현한다.
2. 현업에서 통용되는 전문 용어로 표현한다.
3. 관계 이름은 DB 안에서 유일해야 한다.
4. 가능하면 개체 간의 관련성을 나타내는 동사로 표현한다.

- 답: **2번**
- 설명: 교안은 관계 이름을 부연 설명이 필요 없는 일반적인 용어로 표현하라고 설명한다.

### 9. 식별자에 대한 설명이 잘못된 것은?
1. 식별자는 개체가 존재하는 동안 값이 변경될 수 있다.
2. 식별자는 반드시 유일한 값을 가져야 한다.
3. 개체의 특성을 대표할 수 있는 식별자를 기본키로 결정한다.
4. 모든 개체는 반드시 하나 이상의 식별자를 포함해야 한다.

- 답: **1번**
- 설명: 식별자는 개체가 존재하는 동안 값이 변경되지 않아야 한다.

### 10. 속성 이름을 결정할 때의 유의 사항이 잘못 설명된 것을 모두 고르시오.
1. DB 안에서는 유일한 이름을 부여해야 한다.
2. 속성 이름 앞에 개체 이름이 포함되지 않아야 한다.
3. 현업에서 익숙한 표준용어를 사용한다.
4. 유지보수를 위해 전체 이름보다는 약어를 사용한다.

- 답: **1번, 2번, 4번**
- 설명:
  - 1번: 속성 이름은 한 개체 또는 관계 안에서 유일하면 되고, 소속이 다르면 같은 이름을 사용할 수 있다.
  - 2번: 개체 이름을 반드시 빼야 하는 것은 아니고, 되도록 포함하지 않게 하는 것이 좋다.
  - 4번: 교안은 약어보다 전체 이름 사용을 권장한다.

### 11. 개념적 설계 단계에서 아래 ER 다이어그램을 작성한 다음에 정의한 제약사항으로 적절한 것은?
1. 각 문화센터의 센터장은 1명이다.
2. 강좌마다 강의실이 1개씩 배정된다.
3. 강좌명이 동일한 강좌라도 개강 일자가 다르면 새로운 강좌로 본다.
4. 강좌명이 동일한 강좌는 여러 번 개설될 수 있으나 한 분기에는 한 번만 개설된다.

- 답: **4번**
- 설명: 4번은 ER 다이어그램만으로 직접 표현하기 어려운 운영상의 제약사항, 즉 업무 규칙에 해당한다.

### 12. 트랜잭션 모델링의 필요성에 대한 설명이 잘못된 것은?
1. DB 설계 초기에 주요 트랜잭션의 기능적 특성을 파악할 수 있다.
2. 추후 구현 단계에서 트랜잭션을 처리하는 응용 프로그램을 작성하는 기초 자료가 된다.
3. 데이터의 구성과 표현을 파악할 수 있다.
4. 트랜잭션에 필요한 모든 정보가 개념적 스키마에 포함되어 있는지 확인할 수 있다.

- 답: **3번**
- 설명: 데이터의 구성과 표현을 파악하는 것은 개념적 스키마 모델링의 목적에 가깝고, 트랜잭션 모델링은 데이터 처리에 초점을 둔다.
